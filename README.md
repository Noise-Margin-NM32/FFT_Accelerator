# 🛡️ Kavach 512-Point FFT Multi-Slave SoC Interconnect

Welcome to the official repository for the **Kavach 512-Point FFT Multi-Slave SoC Interconnect**. This project integrates a high-performance **1 TOPS FFT pipeline accelerator** and a **16KB zero-wait-state Scratchpad SRAM memory** onto a central **AMBA AHB (v2.0)** bus system, managed dynamically by a central AHB Arbiter & Address Decoder.

---

## 📐 Section 1: SoC System Architecture

The SoC integrates multiple SystemVerilog/Verilog Soft IP blocks onto a shared AMBA AHB bus, ensuring cycle-accurate decoding, wait-state management, and collision-free boundary transitions.

```
       +-------------------------------------------------------------+
       |                         AHB Master                          |
       +-------------------------------------------------------------+
                                      |
                                      v
       +-------------------------------------------------------------+
       |               AHB Arbiter & Address Decoder                 |
       +-------------------------------------------------------------+
                   |                                   |
           (HSEL0: 0x4000)                     (HSEL1: 0x0000)
                   v                                   v
       +-----------------------+           +-----------------------+
       |  FFT Wrapper & Core   |           |    16KB Scratchpad    |
       |  (Slave 0: 4KB space) |           |  SRAM (Slave 1: 16KB) |
       +-----------------------+           +-----------------------+
```

### Integrated Soft IP Blocks
1. **AHB Interconnect & Decoder (`ahb_arbiter.v`):** A converted and modernized open-source SystemVerilog core that handles dynamic address decoding, master requests (up to 15), slave responses (up to 15), and contains a default slave to gracefully return bus errors if out-of-boundary addresses are requested.
2. **Scratchpad RAM Controller (`ahb_sram.v`):** A custom parameterizable memory block containing four byte-wide synchronous RAM instances. It supports independent byte, half-word, and word writes using dynamically generated byte-lane write-enables.
3. **Kavach FFT Wrapper (`nm32_fft_ahb_wrapper.v`):** Translates the raw synchronous signals of the 512-point pipeline FFT core to clean AHB-Lite bus specifications, including 1-cycle read stalls to support synchronous memory reads.

## ⏱️ Section 2: The Silicon Timeline

This cycle-accurate timeline documents the electrical transitions and data flows of the SoC from power-on reset to frequency harvesting:

### 1. Power-On Reset (The Clean Slate)
* **Action:** The system controller pulls `hresetn` low.
* **Transition:** The Master FSM enters `IDLE` (State 0). The `done` flag is pulled low, internal address buffers are wiped, and both `hsel` lines are de-asserted.

### 2. Boot & SRAM Check (Populating the Warehouse & Twiddles)
* **Action:** The Master writes sequential test data into the Scratchpad SRAM (`0x0000_0000` to `0x0000_3FFF`). It also simulates the CPU reading Twiddle Factors from SPI Flash and writing them to the FFT's **Twiddle RAM**.
* **Logic:** The Arbiter decodes addresses under `0x4000` to `slv_hsel[1]` (SRAM). Addresses for Twiddles (`0x4800` to `0x4BFC`) route to `slv_hsel[0]` and are stored inside the FFT's internal 256x32 Twiddle RAM.

### 3. Loading the FFT Accelerator (Passing the Baton)
* **Action:** The Master loads 512 complex time-domain samples in **bit-reversed order** into the FFT Data RAM.
* **Logic:** The Master performs 512 consecutive AHB writes to `0x0000_4000` through `0x0000_47FC`. The Arbiter decodes this and routes all data to `slv_hsel[0]`.

### 4. Triggering the Butterfly Math Engine
* **Action:** The Master triggers the computation by writing `1` to the FFT Control Register at `0x0000_4C00` (Bit 0 = `START`).
* **Logic:** The FFT core decouples its data memories from the AHB bus and begins Stage 1 of its Decimation-in-Time (DIT) algorithm, using the CPU-loaded constants from the Twiddle RAM.
* **Done Interrupt:** The Master can either poll the Control Register, or simply wait for the newly integrated **hardware `fft_irq` wire** to fire, which directly signals the CPU's Interrupt Controller that processing is complete.

### 5. Harvesting the Spectrum
* **Action:** The Master reads 512 frequency-domain bins starting at `0x0000_4000` and writes them to a output file (`fft_out_soc.txt`) in **natural order**.

## 📊 Section 3: Pinout & Register Specifications

### 1. Central SoC Wrapper (`nm32_soc_top`)

| Port Name | Direction | Width | Type | Description |
| :--- | :---: | :---: | :---: | :--- |
| `hclk` | Input | 1 | wire | Central system clock (Target: 180 MHz). |
| `hresetn` | Input | 1 | wire | Central Active-Low Synchronous Reset. |
| `m_hbusreq` | Input | 1 | wire | Master Bus Request. Asserted when Master wants to drive. |
| `m_hlock` | Input | 1 | wire | Master Locked Transfer Strobe. |
| `m_htrans` | Input | 2 | wire | Master Transfer Type (Idle, Busy, NonSeq, Seq). |
| `m_haddr` | Input | 32 | wire | Master Address Bus. Mapped to 0x0000–0x4FFF. |
| `m_hwrite` | Input | 1 | wire | Master Write Control (1 = Write, 0 = Read). |
| `m_hsize` | Input | 3 | wire | Master Transfer Size (000 = Byte, 001 = Half-Word, 010 = Word). |
| `m_hburst` | Input | 3 | wire | Master Burst Type. |
| `m_hprot` | Input | 4 | wire | Master Protection Control. |
| `m_hwdata` | Input | 32 | wire | Master Write Data Bus. |
| `m_hgrant` | Output | 1 | wire | Master Bus Grant. Asserted by Arbiter when Master gains the bus. |
| `m_hready` | Output | 1 | wire | Master Ready feedback (indicates transfer has completed). |
| `m_hresp` | Output | 2 | wire | Master Bus Response status (00 = OKAY, 01 = ERROR). |
| `m_hrdata` | Output | 32 | wire | Master Read Data Bus (fed from selected slave). |
| `fft_irq` | Output | 1 | wire | Hardware Interrupt signal; fires when FFT calculation finishes. |

## 🔄 Section 4: Dynamic Co-Simulation Verification

We have fully automated the co-verification between **MATLAB** (which acts as our signal generator and analysis console) and **Vivado** (which simulates the RTL logic).

### Instructions for running:
1. **Open MATLAB:** Run `NM32_KAVACH.m`. It will dynamically locate its own path, generate `audio_in.txt` directly into the project and simulation directories, and pause.
2. **Execute Vivado Simulation:** In your Vivado Tcl console, run:
   ```tcl
   reset_simulation
   launch_simulation
   run -all
   ```
   *The testbench reads the input stimulus, executes all multi-slave bus sequences, and writes `fft_out_soc.txt`.*
3. **Render Plots:** Go back to MATLAB and press any key to resume. MATLAB will load the output, reconstruct the magnitudes, and plot a cycle-accurate side-by-side comparison between the floating-point golden model and your SystemVerilog silicon output!
